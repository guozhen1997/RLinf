GR00T-N1.6 Model Reinforcement Learning Training
==================================================================

.. |huggingface| image:: /_static/svg/hf-logo.svg
  :width: 16px
  :height: 16px
  :class: inline-icon

This example provides a complete guide on how to fine-tune the GR00T-N1.6 algorithm using reinforcement learning in the LIBERO environment with the RLinf framework. It covers the entire process from environment setup, core algorithm design, training configuration, evaluation, and visualization, along with reproducible commands and configuration snippets.

The main goals are to develop a model capable of performing robotic operations, specifically including:

1. **Vision and Language Understanding**: Utilize a large-scale pre-trained frozen backbone to process multi-view RGB images and natural language task instructions.
2. **Smooth Continuous Control**: An action head based on flow matching (Flow Matching / Diffusion) architecture to generate high-frequency, smooth continuous action trajectories.
3. **Reinforcement Learning Fine-Tuning**: Optimize the policy through the PPO algorithm combined with environmental feedback.

Environment Description
------------------------------------------------------------------

**LIBERO Environment**

- **Environment**: The LIBERO simulation benchmark built on robosuite (MuJoCo).
- **Tasks**: Control a 7-degree-of-freedom robotic arm to perform various household manipulation skills (pick-and-place, stacking, opening drawers, spatial rearrangement).
- **Observations**: RGB images captured by off-screen cameras placed around the workspace (typical resolutions of 128×128, 224×224, or 256×256).
- **Action Space**: The environment natively provides 7-dimensional continuous actions—3D end-effector position control (x, y, z), 3D rotation control (roll, pitch, yaw), and gripper control (open/close). *Note: GR00T-N1.6 will zero-pad these 7-dimensional actions to a 128-dimensional cross-embodiment universal action space at the bottom layer via embodiment tags.*

**Task Description Format**

GR00T-N1.6 directly uses the natural language task descriptions provided by the environment as input to the language model, utilizing the frozen backbone network to extract semantic features.

**Data Structure**

- **Images**: Continuous RGB video frames from the main view and wrist view, typically named ``main_images`` and ``wrist_images``. Considering timestep history, the shape is usually ``[batch_size, seq_len, 224, 224, 3]``.
- **State**: End-effector position, pose, and gripper state (concatenated with visual features at the network bottom as state representation).
- **Task Description**: Natural language instructions.
- **Rewards**: Sparse rewards for PPO reinforcement (1 for success, 0 for failure).

Algorithm Components
------------------------------------------------------------------

**Core Algorithm Components**

1. **PPO (Proximal Policy Optimization)**

  - Use GAE (Generalized Advantage Estimation) for advantage estimation.
  - Policy clipping with ratio limits.
  - Value function clipping.
  - Entropy regularization.

Environment and Cross-Embodiment Support
------------------------------------------------------------------
**Flexible Embodiment Ecosystem and Modality Config**

GR00T-N1.6 adopts a highly decoupled data interface design. By configuring different ``Embodiment Tags`` and modality converters, the framework not only natively supports 7-degree-of-freedom robotic arms (3D position, 3D rotation, gripper) in environments like LIBERO, but can also seamlessly extend to quadruped robot morphologies through the same underlying continuous control parameters, without any invasive modifications to the underlying network architecture.

GR00T-N1.6 Core Architecture Upgrades
------------------------------------------------------------------
**1. FSDP (Fully Sharded Data Parallel) Actor Worker**

Given the significant increase in GR00T-N1.6's parameter scale, RLinf's Actor node has been fully refactored into ``EmbodiedFSDPActor``. This architecture shards model weights, gradients, and optimizer states across multiple GPU nodes, breaking the single-GPU memory bottleneck of traditional DDP and greatly improving throughput.

**2. Head-only Fine-Tuning and Explosion Protection (Gradient Protection)**

To adapt to alignment with physical environments using small amounts of data and save computational resources, GR00T-N1.6's RL training employs a partial fine-tuning strategy:

- **Backbone Freezing**: Strictly lock the billions of parameters in the vision and language backbone networks, running them only in inference mode (Eval).
- **Action Head Fine Control**: Only unfreeze the action output head (Action Head) to learn physical dynamics. Furthermore, the framework forcibly freezes the timestep encoder (``timestep_encoder``) and intercepts gradient hooks, precisely preventing ``NaN`` gradient explosions that are prone to occur in high-dimensional action distributions.

GR00T-N1.6 Core Architecture Design Philosophy
------------------------------------------------------------------
Here, we introduce the core architecture upgrades and important design philosophies of GR00T-N1.6 to help developers understand the underlying logic more deeply.

**1. Two-Stage Decoupled Training Paradigm**

The RLinf framework adopts a highly decoupled two-stage training architecture for GR00T-N1.6:

- **First Stage (Pure SFT Warm-up)**: Use ``Pure SFT Model`` mode. The model is completely detached from the physical simulation environment, relying only on offline expert datasets for supervised fine-tuning, focusing on fitting target action trajectories.
- **Second Stage (PPO Reinforcement Alignment)**: Based on SFT convergence, load the model into the FSDP-based distributed Actor for real-time interaction with the simulation environment.

**2. Minimal Local Fine-Tuning Strategy (Head-only Fine-Tuning)**

To save memory while preventing "catastrophic forgetting," the framework defaults to a "frozen backbone" strategy:

- **Backbone Freezing**: In both SFT and subsequent RL stages, the vision-language backbone network is strictly locked (``requires_grad=False``).
- **Focus on Action Head**: Only unfreeze the action output head (Action Head) for gradient updates.

**3. Flow-Matching Action Generation**

- The model generates high-frequency action chunks directly in continuous space through a flow-matching mechanism (Flow-SDE / Diffusion) of adding noise and denoising.
- Key Configuration: Control prediction step size via ``num_action_chunks``, and denoising depth via ``denoising_steps``.

**4. Cross-Embodiment Generalization**

- **Embodiment Tags**: Relying on passed configuration tags (e.g., ``ROBOCASA_PANDA_OMRON``), the system can dynamically adapt to corresponding state encoders and action spaces. Whether it's a single-arm robotic arm or a quadruped robot morphology, it can be reused.

**5. FSDP Distributed Parallel Architecture**

- The underlying system has refactored the Actor node (e.g., ``EmbodiedFSDPActor``) to shard model weights, gradients, and optimizer states across GPU nodes.

Running Scripts and Configuration
------------------------------------------------------------------

**1. Key Cluster Configuration**

.. code-block:: yaml

  cluster:
    num_nodes: 1
    hardware_ranks: [[0]]
    component_placement:
     actor,env,rollout: all

**2. Key Model Parameter Configuration**

**2.1 Actor Model and Action Head Configuration (`actor.model`)**

.. code-block:: yaml

  model:
    model_type: "gr00t_1_6"   
    add_value_head: True          # RL Key: Dynamically inject value network for advantage prediction
    num_action_chunks: 1          # Set to 1 in PPO stage to improve real-time performance for high-frequency physical interactions
    denoising_steps: 1            # Control flow-matching denoising steps

**2.2 FSDP Distributed Sharding Strategy**

.. code-block:: yaml

  no_split_modules:
    - "Qwen3DecoderLayer"
    - "Siglip2EncoderLayer"
    - "FlowMatchingActionHeadForRLActionPrediction"
    - "TimestepEncoder"
    - "ValueHead"

**2.3 PPO and Optimizer Hyperparameters**

.. code-block:: yaml

  algorithm:
    adv_type: gae
    clip_ratio_high: 0.2
    gamma: 0.99
    gae_lambda: 0.95

  optim:
    lr: 5.0e-6
    value_lr: 1.0e-4
    clip_grad: 1.0

**3. Configuration File Paths**

- GR00T-N1.6 + PPO + Libero-Spatial:
  ``examples/embodiment/config/libero_spatial_ppo_gr00t_16.yaml``

Modify the path to the post-SFT model:

.. code-block:: yaml

  model:
    model_path: "/path/to/GR00T-N1.6-libero-spatial-ppo"

**4. Launch Command**

.. code-block:: bash

  bash examples/embodiment/run_embodiment.sh libero_spatial_ppo_gr00t_16

Visualization and Result Monitoring
------------------------------------------------------------------

**1. TensorBoard Logging**

.. code-block:: bash

  tensorboard --logdir ./logs --port 6006

**2. Key Monitoring Metrics**

- **Training Metrics**: ``actor/loss``, ``actor/value_loss``, ``actor/grad_norm``, ``actor/approx_kl``.
- **Rollout Metrics**: ``rollout/returns_mean``, ``rollout/advantages_mean``.
- **Environment Metrics**: ``env/episode_len``, ``env/success_once``.

**3. Video Generation and Integration Configuration**

.. code-block:: yaml

  video_cfg:
    save_video: True
    video_base_dir: ${runner.logger.log_path}/video/eval

  runner:
    logger:
     logger_backends: ["tensorboard"]

After fine-tuning, the system will generate ``metadata.json`` and other statistical files in the output directory, preserving key modality information needed for inference and subsequent deployment.
